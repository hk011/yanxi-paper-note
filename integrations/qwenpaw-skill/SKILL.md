# 研析（Yanxi）论文解读 Skill

## 定位

本 Skill 提供**研析笔记生成方法论**（流程 + Prompt 规范），不要求启动研析后端、不要求 `YANXI_API_KEY`。

Agent 应使用**自身已有工具**完成：

| 步骤 | Agent 工具示例 |
|------|----------------|
| 读取 / 解析 PDF | 文件读取、PDF 解析、OCR、代码执行等 |
| 联网补充背景 | `web_search`、浏览器等 |
| 生成配图 | 文生图工具（可选） |
| 写出笔记 | 大模型写作 |
| 交付 | 将 `note.md` 转为 PDF，`send_file_to_user` 发送 `.pdf` |

> **与 Web 端研析的差异**：无 MinerU VLM 精细解析时，图表提取可能较弱；笔记**结构与写作规范**与研析 Web 端一致。

---

## 总体流程

```
① PDF 提取          ② 结构化整理         ③ 三阶段笔记生成        ④ 转 PDF 并交付
──────────────────────────────────────────────────────────────────────────────
读 PDF 全文      →  paper_skeleton   →  阶段一：解读大纲     →  note.md → PDF
提取图片+图题       image_catalog         阶段二：六章并行起草      send_file_to_user
保存到工作目录                            阶段三：综合重写终稿
```

**工作目录建议**（与 PDF 同目录或用户指定）：

```
paper_work/
  source.pdf
  parsed.md          # 论文 Markdown 译文/结构化正文（含 ![](images/xxx.jpg)）
  images/            # 从 PDF 提取的图表
  assets/            # Agent 文生图生成的辅助讲解图（可选）
  outline.txt        # 阶段一大纲（中间产物，可选保留）
  note.md            # 笔记终稿（中间产物）
  note.pdf           # 最终交付物（由 note.md 转换）
```

## ① PDF 文字提取与结构化

### 目标

得到两份材料，供后续笔记生成使用：

1. **`parsed.md`**：论文结构化 Markdown
   - 保留章节层级（`#` / `##` / `###`）
   - 公式用 `$...$` 或 `$$...$$`
   - 表格用 GFM 表格
   - 图片：`![Figure 1: 图题说明](images/文件名.jpg)`（路径必须真实存在）

2. **`image_catalog`（图片清单）**：每张图一条记录

```text
- images/fig1.jpg | Figure 1 | 模型整体架构示意图 | p3
- images/fig2.png | Figure 2 | 各数据集上准确率对比 | p7
```

### 提取要求

- **全文覆盖**：摘要、引言、方法、实验、结论、参考文献要点
- **专业术语**：首次出现可保留英文并附中文，如 Transformer（变换器）
- **图题绑定**：每张图必须有 Figure 编号 + 中文图意说明；禁止编造不存在的图片路径
- **页码标注**（可选）：段落前加 `[p3]` 便于溯源

### `paper_skeleton` 格式

将 `parsed.md` 压缩为模型可读的骨架（建议 ≤12000 字）：

```text
[p1] text: Abstract — ...
[p2] title: Introduction
[p2] text: ...
[p3] figure: Figure 1 Architecture of ...
[p4] table: Table 1 Results on ...
```

若 Agent 无法精细分块，可直接截取 `parsed.md` 前 12000 字作为 skeleton。

### `image_list` 文本格式（注入 Prompt）

```text
1. images/abc123.jpg → Figure 1: 模型架构图（p3）
2. images/def456.jpg → Figure 2: 实验结果对比（p7）
```

## ② 笔记生成：系统 Prompt（全程使用）

每次调用大模型写笔记时，**System** 内容：

```text
你是一位资深科研论文解读专家，擅长将英文学术论文转化为结构化、通俗易懂的中文 Markdown 解读笔记。

要求：
1. 使用中文撰写，专业术语可保留英文并附中文解释
2. 输出标准 Markdown，不要包裹在代码块中
3. 引用论文原图时使用 Markdown 图片语法，路径必须从「可用图片清单」中选择，且必须与清单中标注的 Figure/图题一致
4. 需要补充背景、综述、解读博客、GitHub 代码等信息时，主动联网检索
5. 搜索优先：概念背景、相关综述、论文解读博客、GitHub 开源仓库
6. 不要编造不存在的链接；引用搜索结果时标注来源
7. 表格必须使用标准 GFM Markdown：表格前后各留一个空行，表头、分隔行、数据行独占一行；不要把表格写进列表项、段落、引用块或同一行文本中；表格单元格内不要使用裸竖线，必要时用"/"替代
8. 不要使用 emoji 或特殊彩色符号表示是/否，表格中统一使用"是/否/部分"

【配图原则】
- 优先引用论文原图（images/ 路径）
- 仅当「画一张图能明显帮助读者理解当前内容」时，再调用文生图工具生成辅助图
- 生成图保存到 assets/，笔记中用 ![](assets/文件名.png) 引用，并注明「AI 生成示意图」
- 文生图提示词：16:9 横构图，扁平矢量插画，色彩明快，图内中文文字用双引号逐字写出，数值须具体
```

若 Agent **不支持联网**，将第 4–6 条替换为：

```text
4. 仅依据论文原文、解析内容与已有知识撰写
5. 禁止调用联网搜索；需要外部背景时写「可进一步检索」而非虚构链接
6. 不要编造不存在的 URL 或引用来源
```

## ③ 三阶段笔记流水线

与研析 Web 端 `note_pipeline` 一致，**必须按顺序执行**，不可跳过阶段一直接写终稿。

### 阶段一：解读大纲

**User Prompt 模板**（替换 `{paper_skeleton}`、`{image_list}`）：

```text
请根据以下论文结构化解析内容，输出「解读大纲」（纯文本，不要完整笔记）：

1. 论文基础信息表（标题中英文、作者、单位、期刊/会议、发表时间、代码/数据链接——能从文中推断则填，否则标「未提及」）
2. 章节大纲（按论文实际结构）
3. 关键概念列表（中英文，标注哪些概念可能需要联网补充背景）
4. 重要图片清单（从正文中识别 Figure 编号 + 说明；后续章节须按清单中的路径引用，不要自行猜测文件名）

【可用图片清单】（含路径绑定，大纲阶段请据此整理图片清单）
{image_list}

---
{paper_skeleton}
```

输出保存为 `outline.txt`（或仅在上下文中传递给阶段二）。

---

### 阶段二：六章并行起草

对以下 **6 个章节分别调用一次**大模型（可并行）。每章**独立撰写**，只输出该章 Markdown（含 `##` 标题）。

| 章节 | 标题 | 撰写要点 |
|------|------|----------|
| 1 | 一、论文基础信息 | Markdown 表格：标题（原文+译文）、作者、单位、期刊/会议、发表时间、代码库、数据集等 |
| 2 | 二、背景、动机与结果 | 核心贡献总结、研究背景与动机、主要结果概览；需要背景时联网搜索 |
| 3 | 三、核心方法 | 技术架构详解、关键算法、创新点；**必须详细描述架构图/方案图**（优先引用原图）；可文生图辅助 |
| 4 | 四、实验结果 | 实验设置、主要结果分析（引用论文图表并解读）、对比实验（可用表格） |
| 5 | 五、总结与展望 | 研究价值、局限性、未来方向 |
| 6 | 六、扩展阅读 | 相关论文推荐、参考资料与链接（可联网搜索补充） |

**每章 User Prompt 模板**（替换占位符）：

```text
你正在为一篇论文撰写中文解读笔记。

【已完成大纲】
{outline}

【论文结构化内容摘要】
{paper_skeleton}

【可用图片清单】（每项已绑定图题/类型与路径；引用时必须使用箭头右侧的路径，格式：![](images/文件名.jpg)）
{image_list}

【当前任务】
{section_instruction}

请直接输出本章 Markdown 内容（含章节标题如 ## 一、...）。本章独立撰写，无需参考其他章节草稿。
```

`{section_instruction}` 取上表对应章节的「撰写要点」全文。

**第三章节**是配图重点：若原图不足以说明，调用文生图工具，参考下方「配图 Prompt 规范」。

---

### 阶段三：综合重写终稿

将六章草稿**整体重写**为一篇连贯笔记（禁止机械拼接）。

**User Prompt 模板**：

```text
请基于前面各阶段得到的资料与草稿，输出一篇完整、连贯、图文并茂的中文论文解读笔记。

【论文标题】
{paper_title}

【论文解读大纲】
{outline}

【分阶段草稿材料】（这些只是材料，不要机械拼接；请整体重写成一篇连贯笔记）
{section_drafts}

【论文结构化内容摘要】
{paper_skeleton}

【可用论文原图】（每项已绑定图题/类型与路径；引用时必须使用箭头右侧的路径）
{image_list}

【已生成辅助讲解图】（引用格式：![](assets/文件名.png)）
{generated_images}

输出要求：
1. 只输出最终 Markdown 正文，不要解释过程，不要包裹代码块。
2. 文章结构必须包含：论文基础信息、背景/动机/结果、核心方法、实验结果、总结展望、扩展阅读。
3. 图文并茂：在合适位置插入论文原图和已生成配图（assets/ 路径），并在图下方简要说明图意；AI 生成的配图需注明为示意。
4. 语言要连贯，避免"第一段草稿/第二段草稿"这种拼接感。
5. 所有外部资料链接要标注来源，不要编造来源。
6. 所有 Markdown 表格必须是标准 GFM 表格：表格前后必须空一行，表头、分隔线、每行数据都独占一行；不要在列表项里直接插入表格。
7. 表格内避免使用 emoji、HTML、未转义的竖线。是/否统一写"是""否""部分"。
```

将输出写入 **`note.md`**，进入下方 **⑤ Markdown → PDF** 步骤后再交付。

## 配图 Prompt 规范（文生图，可选）

仅在原图不够直观时生成。固定 **16:9** 横构图。

提示词须为**连贯中文段落**（建议 600–1500 字），按此结构组织：

1. **整体概览**：主标题（双引号写出）、副标题、风格（扁平矢量插画、色彩明快）、背景色、主辅色、构图
2. **模块描述**：各区块位置（左上角/中央/右侧等）、模块标题与正文（双引号逐字写出）、图标与数据（数值须具体，禁止「很多」「较高」等模糊词）
3. **连接与导航**：箭头、虚线、流程方向
4. **装饰与氛围**：背景纹理、阴影等

**图内文字规则**：每一处图内文字用中文双引号 `""` 包裹并逐字写出，如 主标题"数据采集"、标注 "准确率 92%"。

**图类型选型**：

| 内容类型 | 推荐图类型 |
|----------|------------|
| 模型/网络结构 | 模型架构图，模块分层，箭头表数据流 |
| 算法步骤 | 算法流程图，矩形步骤 + 菱形判断 |
| 方法对比 | 左右并列对比图，差异高亮 |
| 原理机制 | 中心机制 + 输入输出因果箭头 |
| 系统流水线 | 多阶段方框左→右连接 |
| 公式推导 | 教学板书图，黑板背景分步展示 |
| 通用要点 | 学术信息图，3–5 分区各含图标与标签 |

## ④ 交付规范

**用户最终收到的文件必须是 PDF**（`{论文简称}_yanxi_note.pdf`），不是裸 Markdown。

流程：`note.md`（写完）→ 转 PDF（本节约束）→ `send_file_to_user` 发送 PDF。

`note.md` 可保留在工作目录作中间产物，但**不要**只把 `.md` 当作最终交付物。

---

## ⑤ Markdown → PDF（必须执行）

### 转换前检查

1. `note.md` 与 `images/`、`assets/` 在**同一工作目录**（或子目录），笔记中的图片路径为相对路径，如 `![](images/fig1.jpg)`
2. 打开 `note.md` 确认每张图路径对应的文件**真实存在**
3. 表格、标题、列表在 Markdown 中格式正确（见上文 GFM 规范）

### 推荐转换方式（按优先级）

Agent 用 `execute_shell_command` 或等价工具，任选一种能**内嵌图片**的方案：

**方式 A：pandoc（推荐，跨平台）**

```bash
pandoc note.md -o note.pdf --pdf-engine=xelatex -V CJKmainfont="Microsoft YaHei" --resource-path=.:images:assets
```

Windows 若无 xelatex，可试：

```bash
pandoc note.md -o note.pdf --resource-path=.:images:assets
```

**方式 B：Python（markdown + weasyprint / pdfkit）**

```bash
pip install markdown weasyprint
# 先将 md 转 HTML（图片用相对路径），再 weasyprint HTML → PDF
```

**方式 C：Node（md-to-pdf）**

```bash
npx md-to-pdf note.md
```

**方式 D：浏览器打印（兜底）**

1. 将 `note.md` 转为 `note.html`（图片路径保持相对路径）
2. 用无头浏览器打开本地 HTML，`打印 → 另存为 PDF`，勾选「背景图形」

### 转换后验证（必须）

```bash
# 文件头必须是 %PDF
# PowerShell 示例：
Format-Hex -Path note.pdf -Count 4
```

- 文件大小应明显大于纯文本（含图时通常 > 100 KB）
- 用 PDF 阅读器抽查：标题、表格、**图片是否正常显示**（非「图片占位」或空白框）

### 禁止事项

- **禁止**用纯文本/HTML 占位符冒充 PDF（文件头不是 `%PDF`）
- **禁止**在图片路径错误或文件缺失时仍交付 PDF——应先修复 `note.md` 中的路径再转换
- **禁止**跳过 PDF 步骤，只 `send_file_to_user` 发送 `note.md`（除非用户明确要求 Markdown）

### 命名与交付

```bash
# 建议命名
{论文简称}_yanxi_note.pdf

send_file_to_user <PDF绝对路径>
```

### PDF 转换失败时

1. 检查图片路径与 `images/`、`assets/` 是否齐全
2. 换用上述另一种转换方式重试
3. 仍失败时：打包 `note.md` + `images/` + `assets/` 为 zip 交付，并告知用户「PDF 转换失败，请用浏览器打开 zip 内 HTML/MD」——此为兜底，非常规流程

**长论文提示**：阶段二可串行执行；整体任务（含 PDF 转换）可能需数分钟，勿中途用简化摘要代替完整流水线。
